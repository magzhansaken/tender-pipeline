## Топология

```
/opt/tenderview/                     ← боевая папка (runtime)
├── .env                             ← секреты (НЕ в git, остаётся на сервере)
├── app.py, load_results.py          ← веб-приложение (кладёт sync.sh)
├── static/*.html                    ← страницы (кладёт sync.sh по маске)
├── Dockerfile, docker-compose*.yml  ← сборка (кладёт sync.sh)
├── Caddyfile, requirements.txt      ← конфиг (кладёт sync.sh)
├── schema.sql                       ← инициализация БД (кладёт sync.sh)
├── *_loop.sh                        ← cron-обёртки (кладёт sync.sh)
├── *.log                            ← логи воркеров
└── tender-pipeline/                 ← клон git-репозитория
    ├── src/                         ← воркеры пайплайна (cron запускает ОТСЮДА)
    ├── web/, deploy/, loops/        ← источник для sync.sh
    └── ...
```

Два важных следствия:

- **Веб-приложение** живёт в `/opt/tenderview` — туда его раскладывает `deploy/sync.sh` и там пересобирается контейнер. Поэтому изменения в `web/` и `deploy/` применяются **только** после `sync.sh`.
- **Воркеры пайплайна** (`src/`) cron запускает прямо из дерева репозитория `tender-pipeline/src/`. Поэтому изменения в `src/` применяются сразу после `git pull` — **без** `sync.sh` и без пересборки.

> ⚠️ На этом же сервере отдельно работает бот **MetaTrader** (vncserver, не в Docker, не в `/opt/tenderview`). Его ничего из деплоя не касается — **не перезагружать сервер и не трогать строку `@reboot ... vncserver` в crontab.**

## Обычный деплой (рабочий цикл)

Стандартный путь для подавляющего большинства изменений (правка кода веб-приложения, страниц, воркеров).

### Шаг 1 — закоммитить и запушить (ПК, PowerShell)

```powershell
cd "C:\Users\magzh\Desktop\Magzhan Проекты\3. Тендер ++\Olloma\tender-pipeline"
git add -A
git commit -m "краткое описание изменения"
git push
```

Файлы кладутся по их местам в репозитории:

| Что меняли | Куда в репозитории |
|------------|--------------------|
| `app.py`, `load_results.py` | `web/` |
| любые страницы (`index.html`, `cabinet.html`, …) | `web/static/` |
| воркеры пайплайна (`search_verify.py`, `publish.py`, …) | `src/` |
| Dockerfile / compose / requirements / Caddyfile / schema | `deploy/` |
| cron-обёртки | `loops/` |

### Шаг 2 — выкатить на сервере (SSH)

```bash
cd /opt/tenderview/tender-pipeline
git reset --hard HEAD && git pull
bash deploy/sync.sh
```

- `git reset --hard HEAD` — **обязательно**: сбрасывает любые случайные локальные изменения на сервере, иначе `git pull` может упасть с конфликтом.
- `sync.sh` копирует `web/` (включая **все** `static/*.html` по маске — новые страницы подхватываются автоматически), `deploy/`, `loops/` в `/opt/tenderview`, делает loop-скрипты исполняемыми и пересобирает контейнер `app`. `.env` не трогается.

### Шаг 3 — проверить

См. [Проверка после деплоя](#проверка-после-деплоя). И обязательно открой страницу в **режиме инкогнито** или с жёстким обновлением (Ctrl+Shift+R) — браузер агрессивно кэширует HTML/JS.

## Частные случаи

### Изменились только воркеры (`src/`)

Пересборка не нужна — cron запускает их из репозитория:

```bash
cd /opt/tenderview/tender-pipeline
git reset --hard HEAD && git pull
# всё; следующий запуск по cron подхватит новый код
```

### Изменились только переменные окружения (`.env`)

**Меняется значение существующей переменной** (например, новый `ADMIN_PASSWORD`): `sync.sh` не нужен — достаточно отредактировать `.env` и пересоздать контейнер, чтобы он перечитал файл:

```bash
cd /opt/tenderview
nano .env
docker compose up -d --force-recreate app
```

**Добавляется НОВАЯ переменная** (например `CERT_API_URL`): переменной мало просто быть в `.env` — её ещё нужно **пробросить внутрь контейнера** через `deploy/docker-compose.override.yml`. Там перечислены все доп-переменные сервиса `app`:

```yaml
services:
  app:
    environment:
      ADMIN_PASSWORD: ${ADMIN_PASSWORD:-}
      OWNER_EMAIL: ${OWNER_EMAIL:-}
      SESSION_DAYS: ${SESSION_DAYS:-30}
      CERT_API_URL: ${CERT_API_URL:-}
      CERT_API_KEY: ${CERT_API_KEY:-}
```

Порядок для новой переменной:
1. Добавить строку проброса в `deploy/docker-compose.override.yml` (в репозитории), закоммитить, запушить.
2. На сервере дописать значение в `/opt/tenderview/.env`.
3. Выкатить: `git reset --hard HEAD && git pull && bash deploy/sync.sh` (пересоберёт `app` и подхватит и проброс, и значение).
4. Проверить, что переменная дошла внутрь: `docker exec tenderview-app-1 printenv CERT_API_URL` → должна вернуть значение, а не пусто.

> Симптом «переменная есть в `.env`, а `printenv` внутри контейнера пуст» = забыт проброс в `docker-compose.override.yml`.

### Изменилась только статика (HTML/JS)

Всё равно через `sync.sh` (он копирует `static/*.html` и пересобирает образ, в который вшита статика). После — жёсткое обновление браузера.

## База данных и миграции

**Миграции применяются автоматически при старте контейнера `app`** — отдельные SQL-команды руками выполнять не нужно.

- `lots` и `tenders` создаются из `deploy/schema.sql`.
- Таблицы веб-приложения (`users`, `sessions`, `payments`, `metrics_daily`, `favorites`) и функция `tf_region()` создаются идемпотентно при инициализации приложения.
- Новые колонки добавляются через `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` (например, у `favorites` так появились `status`, `note`, `checklist`). Это безопасно для существующих данных — старые строки получают значения по умолчанию.

Поэтому после обычного `sync.sh` (который пересобирает и перезапускает `app`) схема приводится в актуальное состояние сама. Данные при этом не теряются — пересобирается только контейнер приложения, том БД остаётся.

Посмотреть данные напрямую (значения берутся из `/opt/tenderview/.env`):

```bash
docker exec -e PGPASSWORD="$POSTGRES_PASSWORD" tenderview-db-1 \
  psql -U "$POSTGRES_USER" -d "$POSTGRES_DB" -c "SELECT count(*) FROM lots;"
```

## Проверка после деплоя

```bash
# контейнеры живы
docker compose ps           # из /opt/tenderview

# витрина и панель отвечают (ожидаем 200)
curl -s -o /dev/null -w "%{http_code}\n" https://$DOMAIN/
curl -s -o /dev/null -w "%{http_code}\n" https://$DOMAIN/admin

# логи приложения без ошибок старта (схема/миграции применяются здесь)
docker compose logs --tail=40 app

# бот MetaTrader не задет
vncserver -list
```

Затем открой **/admin** → блок «Воркеры» покажет, что каждая задача отрабатывает (после первого запуска по расписанию). Красный воркер — смотри его лог в `/opt/tenderview/<имя>.log`.

> Порт `8000` наружу не открыт — приложение доступно только через Caddy по HTTPS на домене. Тестировать curl'ом нужно по `https://$DOMAIN/...`, а не по `localhost:8000`.

## Cron (расписание воркеров)

Целевой crontab (строку бота `@reboot ... vncserver` **не трогать**):

```cron
@reboot sleep 10 && vncserver :1 -geometry 1280x800 -depth 24
0  */4 * * * /opt/tenderview/daily_sync_loop.sh    # сверка с goszakup + новые лоты + статусы
*/10 * * * * /opt/tenderview/process_loop.sh       # нормализация Ollama
*/15 * * * * /opt/tenderview/search_loop.sh        # поиск на площадках + кандидаты
*/15 * * * * /opt/tenderview/publish_loop.sh       # публикация в витрину
0  *   * * * /opt/tenderview/wb_loop.sh            # добор цен Wildberries
*/30 * * * * /opt/tenderview/alibaba_loop.sh       # добор ориентиров Alibaba
```

Каждый loop-скрипт защищён от двойного запуска (проверяет, не крутится ли уже его контейнер) и сам завершается, обработав свою пачку. Редактирование: `crontab -e`. Перед изменением — бэкап: `crontab -l > /tmp/cron.bak`.

## Откат

Если деплой сломал приложение — откатиться на предыдущий рабочий коммит:

```bash
cd /opt/tenderview/tender-pipeline
git log --oneline -5            # найти хороший коммит
git reset --hard <commit_sha>
bash deploy/sync.sh
```

Полное восстановление из бэкапа (если есть архив):

```bash
cd /opt/tenderview
tar -xzf /путь/к/tenderview_backup_*.tar.gz
docker compose up -d --build app
crontab /tmp/cron.bak
```

Контейнер БД и том с данными при откате приложения не затрагиваются.

## Траблшутинг

| Симптом | Что делать |
|---------|------------|
| Изменения не видны в браузере | Жёсткое обновление Ctrl+Shift+R или режим инкогнито — кэш HTML/JS |
| `git pull` падает с конфликтом | `git reset --hard HEAD` перед `git pull` (на сервере правки не нужны) |
| Воркер красный в `/admin` | Смотри `/opt/tenderview/<имя>.log`; проверь доступность Ollama/площадок |
| `app` не поднимается | `docker compose logs --tail=80 app` — обычно ошибка в коде или подключении к БД |
| Ошибка миграции при старте | Лог `app` покажет проблемный SQL; том БД цел, чинится правкой кода и повторным `sync.sh` |
| Новая страница даёт 404 | Проверь, что есть и роут в `app.py`, и файл в `web/static/`; затем `sync.sh` |
| Изменения в `src/` не применились | Они применяются по cron из репозитория — убедись, что `git pull` прошёл; `sync.sh` для них не нужен |
| `printenv` внутри контейнера пуст, хотя в `.env` есть | Новая переменная не проброшена — добавь её в `deploy/docker-compose.override.yml`, затем `sync.sh` |
| Сертификаты пишут «скоро появится» | `CERT_API_URL` пуст или не проброшен; проверь `docker exec tenderview-app-1 printenv CERT_API_URL` |

## Первичная установка с нуля

> Обычно не требуется — система уже развёрнута. Раздел для восстановления на новом сервере.

1. Установить Docker + Docker Compose, склонировать репозиторий в `/opt/tenderview/tender-pipeline`.
2. Создать `/opt/tenderview/.env` по шаблону `.env.example` (реальные секреты — не из git).
3. Поднять окружение и разложить приложение:

   ```bash
   cd /opt/tenderview/tender-pipeline
   bash deploy/sync.sh        # требует уже существующий /opt/tenderview/.env
   ```

   `sync.sh` прервётся, если `.env` отсутствует — это защита от запуска без секретов.
4. Прописать crontab (см. [Cron](#cron-расписание-воркеров)).
5. Убедиться, что Caddy получил TLS-сертификат и домен резолвится на сервер; проверить по [Проверка после деплоя](#проверка-после-деплоя).

DNS, домен (deSEC) и сертификаты Caddy выпускает/обслуживает автоматически при первом обращении.
